# 数据库设计文档

## 一、数据库概述

### 1.1 设计目标

本数据库设计旨在为数据大屏管理系统提供高效、可靠、可扩展的数据存储方案。设计过程中充分考虑了业务需求的完整性、数据安全性、查询性能以及后期扩展能力。通过合理的表结构设计和索引规划，确保系统在高并发场景下仍能保持良好的响应速度。同时，数据库设计遵循第三范式，消除数据冗余，保证数据的一致性和完整性。

在技术选型方面，主数据库采用MySQL 8.0版本。MySQL作为最流行的开源关系型数据库，拥有成熟的优化器和丰富的功能特性，能够满足企业级应用的存储需求。MySQL 8.0引入了窗口函数、CTE等高级SQL特性，为复杂查询提供了更强大的表达能力。

### 1.2 命名规范

数据库对象命名遵循统一规范，便于理解和维护：

表命名采用小写字母和下划线的组合，以业务模块作为前缀。例如用户相关表使用sys_user作为前缀，模板相关表使用dash_template作为前缀。这种命名方式使得表的功能一目了然，便于数据库管理和团队协作。

字段命名同样采用小写字母和下划线的组合，避免使用保留关键字。关联字段采用外键表名加上id后缀的方式，例如user_id表示用户标识。外键约束名称包含两张表的名称，格式为fk_主表名_从表名。

索引命名采用idx_作为前缀，加上涉及的字段名。唯一索引采用uk_作为前缀。命名规范能够快速识别索引类型和用途。

## 三、索引设计

### 3.1 主键与外键索引

所有表均使用bigint自增作为主键，确保插入性能和数据唯一性。外键字段上创建索引，加速关联查询。

用户表（sys_user）的主键索引为id字段，唯一索引包括username和email字段，确保用户信息的唯一性。

角色表（sys_role）的主键索引为id字段，唯一索引为role_key字段，用于权限检查的快速定位。

用户角色关联表（sys_user_role）创建联合唯一索引（user_id, role_id），防止重复分配角色。

模板表（dash_template）的主键索引为id字段，普通索引包括create_user_id和industry字段，用于查询特定用户的模板或特定行业的模板。

组件表（dash_component）的主键索引为id字段，普通索引包括template_id和component_type字段，用于查询特定模板的组件。

数据源表（data_source）的主键索引为id字段，普通索引包括create_user_id和source_type字段。

### 3.2 查询优化索引

根据业务查询特点，创建以下组合索引：

模板列表查询经常按照创建时间和状态进行筛选，因此创建索引（status, create_time）。同时，查询某个用户创建的模板需要按create_user_id筛选，创建索引（create_user_id, create_time）。

组件查询需要根据模板ID获取该模板下的所有组件，因此已有template_id索引。组件更新操作需要根据组件ID快速定位，创建id索引。

数据源查询根据创建用户筛选数据，创建索引（create_user_id, status）。

日志查询根据时间范围和用户ID进行筛选，创建索引（create_time, user_id）。

### 3.3 全文索引

对于需要全文搜索的场景，如模板名称搜索、接口描述搜索等，可以创建全文索引。

模板表的name字段创建全文索引，支持中文分词搜索。

## 四、视图设计

### 4.1 用户权限视图

创建用户权限视图，汇总用户的所有权限信息：

```sql
CREATE VIEW v_user_permissions AS
SELECT 
    u.id AS user_id,
    u.username,
    p.id AS permission_id,
    p.permission_key,
    p.permission_name,
    p.resource_type
FROM sys_user u
LEFT JOIN sys_user_role ur ON u.id = ur.user_id
LEFT JOIN sys_role r ON ur.role_id = r.id
LEFT JOIN sys_role_permission rp ON r.id = rp.role_id
LEFT JOIN sys_permission p ON rp.permission_id = p.id
WHERE u.status = 1 AND r.status = 1 AND p.status = 1;
```

### 4.2 模板组件视图

创建模板组件视图，方便查询模板的完整信息：

```sql
CREATE VIEW v_template_components AS
SELECT 
    t.id AS template_id,
    t.name AS template_name,
    t.industry,
    t.is_published,
    c.id AS component_id,
    c.component_type,
    c.component_name,
    c.position,
    c.sort_order
FROM dash_template t
LEFT JOIN dash_component c ON t.id = c.template_id
WHERE t.status = 1;
```

## 五、分区设计

### 5.1 日志表分区

操作日志表数据量增长较快，采用按月分区的方式管理：

```sql
ALTER TABLE sys_operate_log 
PARTITION BY RANGE (TO_DAYS(create_time)) (
    PARTITION p202401 VALUES LESS THAN (TO_DAYS('2024-02-01')),
    PARTITION p202402 VALUES LESS THAN (TO_DAYS('2024-03-01')),
    PARTITION p202403 VALUES LESS THAN (TO_DAYS('2024-04-01')),
    PARTITION p_future VALUES LESS THAN MAXVALUE
);
```

分区策略能够显著提升大日志表的查询性能，同时方便历史数据的归档和清理。

### 5.2 历史数据归档

对于超过一年且查询频率较低的历史数据，可以归档到历史表中，保持主表的高性能。

## 六、数据安全

### 6.1 敏感数据加密

数据源连接配置中包含数据库密码、API密钥等敏感信息，存储时进行加密处理：

使用AES-256算法对敏感字段进行加密，密钥存储在环境变量或密钥管理服务中。加密后的字段在应用层解密后使用，数据库中只存储密文。

### 6.2 数据备份

制定定期备份策略：

全量备份每周执行一次，备份整个数据库。
增量备份每天执行，备份自上次备份以来的变更数据。
备份文件加密存储，保留周期为30天。

### 6.3 数据脱敏

对于日志查询等场景，涉及用户敏感信息的字段进行脱敏处理：

手机号码显示前三位和后四位，中间用星号替代。
邮箱地址显示前两位和域名，中间用星号替代。
IP地址只保留最后一段，前三段用星号替代。

## 七、ER图设计

### 7.1 核心实体关系

系统核心实体之间的ER关系如下：

用户（sys_user）与角色（sys_role）通过用户角色关联表（sys_user_role）建立多对多关系。一个用户可以拥有多个角色，一个角色可以分配给多个用户。

用户（sys_user）与模板（dash_template）为一对多关系。一个用户可以创建多个模板，模板记录创建者ID。

用户（sys_user）与数据源（data_source）为一对多关系。一个用户可以创建多个数据源，数据源记录创建者ID。

用户（sys_user）与接口配置（api_config）为一对多关系。

模板（dash_template）与组件（dash_component）为一对多关系。一个模板包含多个组件，组件记录所属模板ID。

模板（dash_template）与接口配置（api_config）为一对多关系。一个模板可以配置多个接口。

数据源（data_source）与接口配置（api_config）为一对多关系。一个数据源可以被多个接口使用。

角色（sys_role）与权限（sys_permission）通过角色权限关联表（sys_role_permission）建立多对多关系。

## 八、数据库配置

### 8.1 连接池配置

MySQL连接池配置建议：

最小空闲连接数：5
最大连接数：20
连接超时时间：30秒
空闲连接超时：10分钟
连接最大生存时间：2小时

### 8.2 字符集配置

数据库字符集统一使用utf8mb4，支持完整的UTF-8编码，包括emoji表情字符。

表字符集：utf8mb4_unicode_ci
排序规则：utf8mb4_unicode_ci

### 8.3 存储引擎

所有表使用InnoDB存储引擎，支持事务和行级锁，保证数据一致性。
